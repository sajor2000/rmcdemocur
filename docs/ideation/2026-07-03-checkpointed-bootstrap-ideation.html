Grounding Context

Treating this as a topic in this codebase — making the Neon bootstrap chain resumable after kills, timeouts, or Ctrl-C.

  • Chain today: db:push → db:seed-frameworks (~614 USMLE + ~100 AAMC + ~40 keywords, each Azure-embedded) → db:seed → db:process (14 docs, multi-stage pipeline each).
  • Recent failure: db:seed-frameworks killed at 120/614 USMLE embeds — zero DB progress despite 2+ minutes of Azure spend (embeddings held in memory until full batch completes).
  • Partial fix already shipped: delete framework tables only after all embeddings finish — still loses in-flight embed work on kill.
  • Constraint: drizzle-orm/neon-http throws on db.transaction() — no interactive transactions without switching to neon-serverless Pool.
  • Existing hooks: parsed JSON cache in data/frameworks/parsed/; processing_jobs table tracks upload pipeline stages; clearDocumentArtifacts makes per-document reprocess idempotent.

Topic Axes

  • Framework embedding durability — survive kill mid-USMLE/AAMC/keyword embed
  • DB write granularity — when rows land in Neon vs stay in memory/files
  • Document processing resume — skip or continue per faculty/self-study guide
  • Orchestrator / operator UX — one command, visible status, safe resume

Ranked Ideas

  • #1 Incremental upsert loop
  • #2 Local embedding cache
  • #3 Bootstrap manifest
  • #4 Skip-processed docs
  • #5 Three-phase framework CLI

#1Incremental embed → upsert loop (stable_id as resume key)

Framework embedding DB write granularity Complexity: Medium

Replace batch-then-bulk-insert with a per-row loop: for each USMLE/AAMC/keyword row, skip if DB already has stable_id with non-null embedding; else embed one text, upsert immediately.

for row in usmle: if already_embedded(stable_id): continue vec = await embed(row.text) upsert usmle_domains ON stable_id
Basis
direct: scripts/seed-frameworks.ts:107-121 embeds entire catalog before any INSERT; today's kill at 120/614 wasted Azure calls.
Rationale
Neon row is the checkpoint — no new tables, no transactions needed. Re-run is naturally idempotent.
Downsides
Slower than bulk delete+insert for full refresh; need UNIQUE on stable_id (or upsert logic). Full catalog replace requires explicit --force flag to delete stale rows.
Confidence
92%

#2Append-only local embedding cache (JSONL sidecar)

Framework embedding Complexity: Low

After each successful Azure embed, append {stable_id, model, dims, vector} to data/frameworks/.embedding-cache.jsonl. On resume, read cache first; only call Azure for missing IDs. Flush to Neon in batches of N.

Basis
direct: parsed JSON already cached at data/frameworks/parsed/ — embeddings are the expensive recomputable artifact.
Rationale
Protects Azure spend even when DB is empty or wiped. Cache survives db:push reset. Gitignore the file.
Downsides
Disk size (~750 × 1536 floats ≈ few MB); must invalidate cache when embed model/dims change.
Confidence
88%

#3Bootstrap manifest + npm run db:bootstrap

Orchestrator UX Complexity: Medium

Track progress in data/bootstrap-state.json:

{ "schema": "done", "frameworks": { "usmle": "614/614", "aamc": "done", "keywords": "pending" }, "course_seed": "done", "processed_document_ids": [1, 2, 5] }

Single orchestrator reads manifest, runs next incomplete step, updates after each chunk. Operator runs one command; Ctrl-C and re-run continues.

Basis
direct: plan 004 KTD-2 strict order + manual 4-command chain; user stopped automated run because they lost control/visibility.
Rationale
Unifies resume semantics across framework + process stages; manifest is human-readable checkpoint log.
Downsides
Manifest can drift from DB truth if someone runs scripts manually — orchestrator should reconcile counts from DB on start.
Confidence
85%

#4Skip already-processed documents in db:process

Document processing resume Complexity: Low

Before runFullPipeline, query: doc has chunks with embeddings AND alignments? If yes and not --force, log skip and continue. Optional --from-document-id=N.

Basis
direct: lib/pipeline.ts clearDocumentArtifacts + processing_jobs already support idempotent re-upload; db:process has no skip logic today.
Rationale
Case 1 smoke then full run becomes safe without re-spending hours on completed guides.
Downsides
Partial failure (chunks but no alignments) looks "done" unless check includes pipeline stage completeness.
Confidence
90%

#5Split framework seed into three resumable CLIs

Orchestrator UX Complexity: Low–Medium

db:seed-usmle, db:seed-aamc, db:seed-keywords — each incremental-upsert its table only. Orchestrator or operator runs them separately; failure in keywords doesn't touch USMLE rows.

Basis
direct: seed-frameworks.ts already has three logical blocks (~614 / ~24 / ~40 rows) with separate embed batches.
Rationale
Coarse checkpoints without new infrastructure; matches mental model of bootstrap phases.
Downsides
More npm scripts; doesn't help within a 614-row USMLE run unless combined with #1.
Confidence
82%

Recommended Combination

Ship #1 + #2 together for framework seed (incremental upsert + JSONL cache). Add #4 for process resume. Layer #3 when you want a single operator command — it should call the incremental scripts, not replace them.

db:bootstrap ├─ push (skip if tables exist) ├─ seed-frameworks --resume [#1 upsert + #2 cache] ├─ seed (skip if course 1 exists) └─ process --skip-complete [#4]

What We Rejected (and why)

  • Migrate all of lib/db.ts to neon-serverless Pool for transactions — heavy infra change; upsert-per-row achieves durability without it.
  • Store embeddings in git — binary bloat; local gitignored cache is enough.
  • Redis / external job queue — overkill for a local demo bootstrap.
  • Keep current batch-in-memory embed — proven failure mode today (120/614 lost).
  • Manual 20-step runbook only — doesn't scale; manifest (#3) automates the same checkpoints.
  • Separate SQLite bootstrap DB — second source of truth; Neon + JSONL/manifest suffices.
